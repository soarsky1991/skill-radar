# GitHub Growth Operating Playbook

Languages: [简体中文](#zh-cn) · [English](#english)

<a id="zh-cn"></a>

## 简体中文

### 定位

每个项目都应该是窄而清晰的工具：README 强、证据持续更新，并且和一个高速变化的生态有明确关系。

目标不是发布很多随机 repo。目标是建立一条可见链路：

```text
research -> report -> proof of demand -> small tool -> examples -> maintenance -> next tool
```

### 仓库检查清单

- 一句话描述清楚。
- README 第一屏出现 demo command。
- 第 3 天前补截图、GIF 或报告 excerpt。
- 配好 topics，方便搜索发现。
- API 稳定之后再设置 `good first issue`。
- 前 14 天每天或隔天更新 report。
- Launch post 要带证据，不只写公告。

### 分发循环

1. 发布每日 report commit。
2. 把前 3 个发现改成短内容。
3. 到 2-3 个相关 GitHub issues 下用中立问题验证。
4. 加入一个来自用户反馈的 example 或 query。
5. 用最好的新证据更新 README。

### Star 目标机制

要到 1,000 stars，repo 需要的不只是实用性，还要有让人反复回来的理由。

可重复访问的 hook：

- 每日 leaderboard。
- 每周 "what changed" report。
- 公开 data JSON。
- 从扫描结果沉淀 curated `awesome` section。
- 兼容多个 agents，而不是只押一个 vendor。
- 简单贡献路径：加 query、加 repo、改 scoring rule。

### 两周节奏

Week 1：证明 radar 能产生有用发现。

Week 2：孵化或加深最强发现。如果没有强发现，就改进 crawler，不强行开新坑。

## English

### Positioning

Each project should be a narrow utility with a strong README, daily evidence, and a clear relationship to a fast-moving ecosystem.

The goal is not to publish many random repos. The goal is to create a visible chain:

```text
research -> report -> proof of demand -> small tool -> examples -> maintenance -> next tool
```

### Repository Checklist

- Clear one-line description.
- Demo command in the first screen of the README.
- Screenshot, GIF, or report excerpt by day 3.
- Topics configured for search discovery.
- `good first issue` only after the API is stable.
- Daily or every-other-day report updates for the first 14 days.
- Launch post with evidence, not just announcement language.

### Distribution Loop

1. Publish a daily report commit.
2. Turn the top 3 findings into a short post.
3. Reply to 2-3 relevant GitHub issues with neutral validation questions.
4. Add one example or query that came from user feedback.
5. Update README with the best new evidence.

### Star Goal Mechanics

For 1,000 stars, the repo needs more than utility. It needs a repeatable reason to revisit.

Useful repeat hooks:

- Daily leaderboard.
- Weekly "what changed" report.
- Public data JSON.
- A curated `awesome` section from the scan.
- Compatible with multiple agents, not just one vendor.
- Easy contribution path: add a query, add a repo, improve a scoring rule.

### Two-Week Cadence

Week 1: Prove that the radar creates useful findings.

Week 2: Spin out or deepen the strongest finding. If no strong finding appears, improve the crawler rather than forcing a build.
